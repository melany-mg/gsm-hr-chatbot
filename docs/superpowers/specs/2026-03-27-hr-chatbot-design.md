# GSM HR Chatbot — Design Spec
**Date:** 2026-03-27
**Project:** General Stamping & Metalworks Multilingual HR Chatbot
**Team:** Notre Dame iNDustry Labs

---

## 1. Project Overview

A multilingual RAG-based HR chatbot that allows GSM employees to ask questions about HR policies and receive answers grounded strictly in provided HR documents. The chatbot reduces walk-in HR questions by supporting employees in English, Spanish, Pashto, Dari, and Bosnian. Answers are always cited to their source document. If the chatbot cannot find an answer, it redirects the employee to the HR department.

## 3. Document Ingestion Pipeline

### Trigger
Manually run: `python backend/app/ingest.py`
Run after dropping files into `backend/documents/`. Each run wipes and rebuilds the Qdrant collection from scratch.

### Supported File Types
- `.pdf` — parsed with `pdfplumber`
- `.docx` — parsed with `python-docx`

### Steps

1. **Wipe** — delete and recreate the Qdrant collection
2. **Load** — scan `documents/` for `.pdf` and `.docx` files
3. **Parse**
   - PDFs: use `pdfplumber` to extract text and tables; tables converted to plain text rows (e.g., `"Plan | Deductible | OOP Max\nHMO | $500 | $3000"`)
   - DOCX: use `python-docx` to extract paragraphs and tables
4. **Chunk** — split text using LangChain `RecursiveCharacterTextSplitter` (500 tokens, 50-token overlap)
5. **Embed** — embed each chunk with `paraphrase-multilingual-mpnet-base-v2` (sentence-transformers)
6. **Store** — vectors + raw chunk text + metadata stored in Qdrant

### Chunk Metadata

```json
{
  "source": "GSM Handbook 2026.pdf",
  "page": 4
}
```

## 4. RAG + Chat Backend

### API Endpoints

| Method | Path         | Purpose                  |
|--------|--------------|--------------------------|
| POST   | `/api/chat`  | Send message, get answer |
| GET    | `/api/health`| Health check             |

### Request / Response

```json
// POST /api/chat — Request
{
  "message": "¿Cuántos días de vacaciones tengo?",
  "language": "es",
  "history": [
    { "role": "user", "content": "..." },
    { "role": "assistant", "content": "..." }
  ]
}

// Response
{
  "answer": "Según el manual de empleados...",
  "citations": [
    { "source": "GSM Handbook 2026.pdf", "page": 4 }
  ]
}
```

`history` is sent from the frontend on every turn. The backend is fully stateless — no sessions stored server-side.

### RAG Flow

1. Embed the user message with `paraphrase-multilingual-mpnet-base-v2`
2. Query Qdrant — retrieve top-5 most relevant chunks
3. Build a prompt:
   - **System:** "Answer only using the provided context. If the answer is not found in the context, tell the employee you don't know and direct them to HR at [HR_CONTACT_PLACEHOLDER]. Respond in [language]."
   - **Context:** concatenated retrieved chunks
   - **History:** prior conversation turns
   - **User:** current message
4. Call LLM (Ollama in dev, Claude in prod)
5. Return answer + citations (source + page from retrieved chunk metadata)

### Anti-Hallucination Strategy

- System prompt explicitly forbids the LLM from using knowledge outside the provided context
- No retrieved context (low similarity score) → response defaults to HR redirect message
- Citations are derived from chunk metadata, not generated by the LLM

### LLM Switching

```python
# config.py
if settings.environment == "development":
    # LangChain OllamaLLM pointing to host.docker.internal:11434
else:
    # LangChain ChatAnthropic with ANTHROPIC_API_KEY
```

The embedding model (`sentence-transformers`) is identical in both environments.

## 5. Frontend (React)

### Components

- **Language Dropdown** — top of page; options: English, Español, پښتو, دری, Bosanski. Selection resets on page refresh (no persistence).
- **Chat Window** — scrollable message history; user bubbles right, bot bubbles left.
- **Citations** — displayed beneath each bot bubble as plain text: *"Sources: GSM Handbook 2026.pdf (p. 4)"*
- **Input Bar** — text input + send button at the bottom.
- **HR Redirect Message** — displayed when chatbot has no answer; static placeholder contact info.

### State (React only, no external library)

- `language` — currently selected language code
- `messages` — array of `{ role, content, citations }` for the current session

Full message history is sent to `/api/chat` on every turn (stateless backend).

### Styling
Plain CSS — no UI component library. Minimal and functional.

### No Authentication
The app is open to anyone who can access it on the network.

## 7. Docker Compose

| Service    | Port | Notes                                                  |
|------------|------|--------------------------------------------------------|
| `qdrant`   | 6333 | Persistent named volume for vector data                |
| `backend`  | 8000 | Depends on `qdrant`; `documents/` mounted as volume    |
| `frontend` | 3000 | Proxies `/api/*` requests to backend                   |

### Ingestion Script
Runs **outside Docker Compose** on the host:
```bash
python backend/app/ingest.py
```
Uses `QDRANT_EXTERNAL_HOST` + `QDRANT_PORT` from `.env` to connect to Qdrant (e.g., `localhost:6333`). This is separate from `QDRANT_HOST=qdrant` used by the backend container, which resolves via Docker internal networking. The `documents/` folder is mounted as a Docker volume so the running backend can reference source metadata.

## 9. Key Constraints

- **No hallucinations:** LLM must only answer from retrieved context. No outside knowledge.
- **No user data stored:** Sessions are stateless. No conversation history persisted.
- **Single embedding model:** `paraphrase-multilingual-mpnet-base-v2` used in both dev and prod — vectors remain consistent.
- **Manual re-indexing:** Documents are not auto-indexed. Each ingest run wipes and rebuilds from scratch.
- **Citations required:** Every answer must include source document name and page number.
